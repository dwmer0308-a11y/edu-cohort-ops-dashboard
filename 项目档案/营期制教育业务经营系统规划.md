# 营期制教育业务经营系统规划

生成日期：2026-06-15

## 一句话定位

这不是单一的书法看板，而是一套面向全 SKU 的营期制教育业务经营系统。

系统核心任务不是“展示数据”，而是回答：

- 每个 SKU 是否健康增长？
- 哪个营期、渠道、销售、课程环节正在漏水？
- 投放成本、商品成本、退费之后，哪些业务真正赚钱？
- 哪些课程内容和销售动作能解释订单高峰？
- 哪些数据已经稳定，可以归档，不必反复调用 Metabase？

## 当前实施范围

当前一期只展开两个你能负责目标、能拿到精准口径的 SKU：

- 书画/书法
- 朗诵

其他 SKU 先作为未来可扩展对象保留在系统设计里，不进入一期页面和指标建设。

原因：

- 书画/书法、朗诵能同时拿到前端、订单、成本、后端课程过程数据。
- 你能制定目标，也能判断指标是否符合业务真实口径。
- 先把两个 SKU 的经营闭环打穿，比一次铺开全公司 SKU 更稳。
- 系统底层仍按多 SKU 设计，未来声乐、钢琴、口琴等可以接入，但不影响一期判断。

## 经营对象

系统应该围绕 7 个经营对象组织，而不是围绕现有页面组织。

### 1. SKU

SKU 是一级经营对象。每个 SKU 都应该有独立经营卡：

- GMV
- 收入
- 商品成本
- 投放成本
- 退费
- 毛利
- Leads
- 加微
- 订单数
- CAC
- CPL
- ROI
- R 值
- 到课、完课、作业
- 后端承接表现

### 2. 营期 Cohort

营期是一批流量、一段教学、一轮转化的共同容器。

统一营期主数据至少需要：

- SKU
- 营期名称
- 营期别名
- 开课日期
- 封板日期
- 开班日期
- 营期阶段
- 前端/后端标记
- 老师/学管/销售归属

### 3. 流量批次

流量批次来自成本与进量明细。

关键字段：

- 线索进量日期
- 平台
- 渠道
- 渠道号
- 创意 id
- 供应商
- 商品类型
- 校正金额
- Leads
- 加微
- 线索成本
- 加微成本
- 重复率
- 性别比例
- 当期成交
- 总成交金额
- total_ROI

### 4. 销售承接

销售不是只看 GMV 排名，而要看承接效率。

关键问题：

- 每个销售承接了多少线索？
- 加微后实际转化多少？
- 价格带能力如何？
- 退费情况如何？
- 同样渠道下谁的人效更高？
- 是否受排班影响？

目前已能从 `1361`、`1728`、`1514` 部分支撑；销售排班和人力成本仍缺。

### 5. 商品与价格

订单表已经能反推每个 SKU 的商品和价格带。

关键字段：

- 商品名称
- 商品 sku
- 支付金额
- 收入
- 商品成本
- 退费金额
- 一级分类
- 二级分类
- 订单分类1
- 订单分类2
- 营期阶段

注意：商品 sku 不是业务 SKU。业务 SKU 是声乐、钢琴、书画、朗诵等。

### 6. 课程交付

后端课程过程表支撑课程交付分析。

关键字段：

- 营期名称
- 学管
- 开班日期
- 课程名称
- 上课日期
- 学员数
- 到课数/到课率
- 完课数/完课率
- 重播数/重播率
- 二次听课数/二次听课率
- 作业提交数/提交率
- 作业点评数/点评率

### 7. 课程内容与话术

课程复盘系统补的是经营链路里最容易缺失的一层：

课程内容动作 → 学员信任/情绪变化 → 支付高峰。

它应成为主系统里的“课程分析”页面，而不是独立孤岛。

核心能力：

- 视频
- 逐字稿
- 订单分钟曲线
- 成交峰值
- 峰值前后话术
- Day1-Day6 课程设计链路
- 销售埋点识别：价格、定金、名额、截止、痛点、收益、背书、案例、异议、行动指令

## 页面规划

### 1. 经营总览

给负责人看。

第一屏只回答：

- 今天/本月全 SKU GMV 是否达成？
- 哪些 SKU 偏离目标？
- 哪些 SKU ROI 或毛利异常？
- 哪些营期需要救？
- 哪些数据已封板，哪些仍在变化？

建议组件：

- 全 SKU 经营矩阵
- 今日异常清单
- 本月收入进度
- 投放成本与毛利摘要
- 需关注营期 Top N

### 2. SKU 作战室

每个 SKU 一张作战台。

模块：

- SKU 经营卡
- 营期列表
- 渠道结构
- 商品价格带
- 成本结构
- 销售承接
- 课程交付
- 历史趋势

书画/书法、朗诵先跑通，后续扩展到声乐、钢琴、口琴、美妆等。

### 3. 流量成本

基于 677。

目标是判断买来的流量是否值得。

核心视角：

- 日期
- SKU
- 营期
- 渠道
- 平台
- 渠道号
- 创意
- 商品类型
- 供应商

关键指标：

- 校正金额
- Leads
- CPL
- 加微
- 加微成本
- 重复率
- 当期成交
- 总成交金额
- total_ROI

### 4. 前端营期

基于 1280、1361、1728。

目标是判断营期转化过程是否健康。

核心视角：

- SKU
- 营期
- 渠道
- 销售
- 工作室
- 分类
- D 阶段

关键指标：

- leads
- 加微
- 进群
- 到课
- 完课
- 作业
- GMV
- R 值
- 转化率
- 退费
- 订单数

### 5. 销售人效

目标不是排名，而是看承接质量。

核心问题：

- 谁更擅长高价产品？
- 谁加微承接强但成交弱？
- 谁成交强但退费高？
- 谁在某类渠道表现更好？
- 排班是否影响结果？

还缺数据：

- 销售排班
- 销售在线时长
- 分配规则
- 人力成本/提成

### 6. 订单产品

基于 1514。

目标是建立产品经营视角。

核心问题：

- 每个 SKU 到底卖什么？
- 核心价格带是什么？
- 商品成本是多少？
- 哪些商品贡献收入？
- 哪些商品退费高？
- 前端课、后端课、电商、权益如何区分？

### 7. 课程分析

基于 110 和课程复盘系统。

目标是把“课程交付”和“课程内容”接到经营结果上。

两个层次：

1. 课程数据分析
   - 到课
   - 完课
   - 重播
   - 二次听课
   - 作业提交
   - 作业点评

2. 课程内容复盘
   - 视频
   - 逐字稿
   - 订单分钟曲线
   - 订单高峰
   - 峰值前话术
   - Day1-Day6 链路完整性

### 8. 数据资产中心

这个页面不是给日常看业务，而是给系统可信度兜底。

它要回答：

- 哪些接口接入了？
- 最近一次更新时间是什么？
- 行数是多少？
- 是否被 2000 行截断？
- 是否已经封板归档？
- 字段是否变化？
- SKU 映射是否完整？
- 哪些数据缺失？

## 数据冻结与快照规划

现在不实施，但系统必须预留这个能力。

### 数据状态

- 实时：营期仍在进行，可重复刷新。
- 待稳定：已封板但可能有 T+1/T+2 数据回流。
- 已封存：复盘确认后固定，不再自动刷新。
- 手动重算：允许覆盖，但必须记录操作人、时间、原因。

### 建议快照颗粒度

- 按接口
- 按 SKU
- 按日期范围
- 按营期
- 按拉取时间

### 快照元数据

每次快照至少记录：

- source_id
- source_name
- query_url
- query_params
- sku
- normalized_sku
- date_range
- fetched_at
- row_count
- column_count
- rows_truncated
- data_watermark
- status
- checksum
- file_path

### 为什么必须有快照

因为同一张 Metabase 表会遇到：

- 原生 CSV 下载失败
- 接口只返回 2000 行
- 同一 SKU 在不同表叫法不同
- 字段可能变
- T+1 数据会回流
- 复盘时需要知道当时看到的数是什么

没有快照层，系统会永远处在“每次打开都重新解释世界”的状态。

## 当前缺口

### 1. SKU 主数据缺口

需要一张统一表：

- normalized_sku
- source_sku
- source_table
- display_name
- owner
- active
- notes

### 2. 营期主数据缺口

前端叫 `营期`，后端叫 `营期名称`，订单叫 `营期名称`，成本表也有 `营期`。

需要统一：

- 营期唯一 ID

- 前端营期名
- 后端营期名
- 订单营期名
- 开课日期
- 封板日期
- 开班日期

### 3. 销售排班缺口

目前能看销售结果，但缺少分母：

- 当日是否排班
- 在线时长
- 分配线索数
- 有效跟进数
- 人力成本
- 提成

### 4. 后端经营缺口

已有课程过程，但还缺：

- 后端订单
- 后端转化
- 续费/扩转
- 学管人效
- 老师课酬
- 服务成本

### 5. 全 SKU 成本完整性

677 成本表里：

- 书法、朗诵、美妆、华彩乐园没有截断
- 声乐、钢琴、口琴触发 2000 行上限

因此未来正式接入时，不能只按 SKU 拉全量，需要按日期或营期切片拉取。

## GitHub 接入建议

GitHub 值得接，但它解决的是工程协作和版本安全，不解决业务口径。

它会让这些事情更容易：

- 每次改动有分支和提交记录。
- 可以用 PR 审查系统重构。
- 可以安全回滚。
- 可以让 Codex 在云端基于仓库上下文工作。
- 可以把规划文档、数据字典、接口说明纳入版本管理。

它不会自动解决：

- SKU 口径不统一。
- Metabase 字段不稳定。
- 数据截断。
- 成本定义混乱。
- 业务是否应该看 GMV、毛利还是 LTV。

建议顺序：

1. 先把规划和数据字典放进仓库。
2. 再接 GitHub。
3. 然后用 PR 的方式分阶段重构。

## 最重要的判断

这个系统的价值不在于“多几个图表”，而在于把原来散在 Metabase、Excel、CRM、课程复盘工具里的东西，统一成一套可追责的经营模型。

最终用户打开系统时，不应该先选报表，而应该先看到：

- 哪个 SKU 在赚钱？
- 哪个 SKU 在烧钱？
- 哪个营期正在坏掉？
- 是流量问题、销售问题、课程问题，还是商品问题？
- 数据是否已经稳定，能不能用于复盘？

这才是全 SKU 版本应该追求的方向。
