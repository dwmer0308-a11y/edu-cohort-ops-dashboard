# ChatGPT 同步物料

生成日期：2026-06-16

## 1. 项目一句话

这是一个面向营期制教育业务的本地/局域网经营看板。当前从“书法项目管理看板”升级为一期聚焦 `书画/书法 + 朗诵` 的经营系统，用来跟踪营收目标、实际达成、营期健康、渠道/销售/课程交付表现，并逐步沉淀可复盘的数据资产。

## 2. 当前项目名称和建议 GitHub 仓库名

当前中文项目名：

- 书法项目管理看板

更准确的产品方向：

- 营期制教育业务经营系统

推荐 GitHub 仓库名：

```text
edu-cohort-ops-dashboard
```

命名理由：

- `edu`：明确是教育业务，不局限于书法单一 SKU。
- `cohort`：对应“营期/批次”的核心经营对象。
- `ops`：强调经营、运营、业务管理，而不是普通数据展示。
- `dashboard`：保留当前产品形态，方便别人一眼知道是看板系统。

备选仓库名：

```text
calligraphy-ops-dashboard
```

适合只想先按“书法项目管理看板”建仓，不急着暴露多 SKU 方向。

```text
cohort-revenue-dashboard
```

适合把一期重点放在营收目标、实际、差额、进度、异常监控。

最终建议：

- 如果这个仓库未来要承载 `书画/书法 + 朗诵 + 更多 SKU`，用 `edu-cohort-ops-dashboard`。
- 如果这个仓库只是先托管现有书法看板，用 `calligraphy-ops-dashboard`。

## 3. 当前业务背景

系统服务的业务是营期制教育转化业务。一个营期通常包含流量接入、加微/进群、行课、转化、追单、封板等阶段。

当前系统最需要回答的问题：

- 本月/今日营收目标是否达成？
- 哪个业务线、SKU、营期落后于目标？
- 目标、实际、差额分别是多少？
- 前端和后端业务分别表现如何？
- 哪些数据已经可靠，哪些还只是临时缓存或样本？
- 未来是否能从当前经营节奏推演出月底结果？

## 4. 一期范围

一期只聚焦两个核心 SKU：

- 书画/书法
- 朗诵

当前不做：

- 全 SKU 展开
- 复杂权限系统
- 全自动 Loop 实现
- 全部课程视频复盘
- 一次性重构旧系统

一期优先做：

- 经营总览
- SKU 作战室
- 数据资产状态
- 营收目标/实际/差额/进度
- 前端/后端拆解
- 可追溯的数据快照和缓存

## 5. 当前系统运行方式

项目是本地 Python HTTP 服务，不是纯静态网页。

本地访问：

```text
http://127.0.0.1:8765
```

启动方式：

```bash
python3 server.py
```

依赖：

- Python 3
- openpyxl
- 浏览器
- 部分 CRM/飞书同步脚本依赖 Node.js

健康检查接口：

```text
GET /api/health
```

## 8. 当前数据源和口径

目标侧：

- 日目标来自飞书目标表。
- 本地缓存写入 `data/revenue-targets.json`。
- 需要关注 `targetSyncedAt`、`targetSource.range`。

实际侧：

- 实际营收来自 CRM 订单数据同步。
- 本地缓存写入 `data/revenue-actuals.json`。
- 实际金额优先信任 CRM 返回的 `totalPriceString`，不要随便手工按行累加替代。

当前业务分段：

- 书法前端
- 书法后端
- 朗诵后端

已知 SKU 映射问题：

- 订单表里可能叫 `书画`
- 成本表里可能叫 `书法`
- 这类映射应进入 SKU 主数据，不应散落在页面逻辑里。

## 9. 关键接口

```text
GET /api/health
```

确认服务是否正常。

```text
GET /api/config
POST /api/config
```

读取和保存本地配置。

```text
GET /api/revenue-overview?month=YYYY-MM
```

读取某月营收目标、实际、日维度、周维度和业务线拆解。

```text
POST /api/revenue-sync
```

从 CRM 更新实际营收缓存。

```text
POST /api/feishu-report
```

发送飞书营收卡片或日报。

```text
POST /api/import-actual
POST /api/import-channel-analysis
```

导入实际营期数据或渠道分析数据。

## 10. v2 规划阶段

阶段 A：业务规划收口

- 确认一期只做 `书画/书法 + 朗诵`。
- 确认页面边界和数据源清单。
- 明确暂不做的内容。

阶段 B：数据优先原型

- 不做纯视觉装饰。
- 从营收目标、实际、差额、SKU、前端/后端开始。
- 先做能支持经营判断的页面原型。

阶段 C：数据资产和接口方案

- 建 SKU 主数据。
- 建营期主数据。
- 设计快照、缓存、字段校验、截断检查。

阶段 D：第一批工程实现

- 在旧系统旁边做 v2 页面。
- 先用本地缓存/样本驱动。
- 不影响旧系统使用。

阶段 E：GitHub 和协作流程

- 建 GitHub 仓库。
- 建分支/PR 流程。
- 明确测试、回滚、数据脱敏规则。

## 11. 给 ChatGPT 的协作要求

如果把这份物料同步给 ChatGPT，希望它按以下方式协作：

1. 先理解业务和现有系统，不要一上来重构代码。
2. 遇到“整体梳理”“重新整合”时，先输出业务地图和阶段方案。
3. 涉及数据口径时，优先确认目标源、实际源、缓存更新时间和字段含义。
4. 涉及设计时，遵循“数据是主视觉，分析是副视觉”。
5. 涉及 GitHub 建仓时，先排除本地日志、缓存、密钥、浏览器登录态和原始数据。
6. 涉及服务异常时，先检查端口、进程和 `/api/health`，不要直接判断为代码 bug。
7. 所有 CRM、飞书、浏览器登录态、`.local/`、原始导出、日志里的敏感信息都不要提交到 GitHub。

## 12. GitHub 建仓前建议检查

建仓前建议先确认：

- `.gitignore` 是否排除 `.local/`、`.codex/`、`.gstack/`、`node_modules/`、`__pycache__/`、`*.pyc`。
- 是否要排除本地运行日志：`*.log`、`*.err.log`。
- 是否要排除本地备份和集成检查报告。
- 是否需要提供一个脱敏样例数据目录，例如 `data/examples/`。
- 是否需要把 `data/config.json` 拆成样例配置和本地私有配置。
- README 是否说明本地启动方式和依赖安装方式。
